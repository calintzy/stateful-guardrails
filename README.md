한국어 · [English](README.en.md)

고객이 챗봇에 보내는 메시지 하나하나는 대개 사소한 불만입니다("또 안 되네요" 정도죠). 그런데 이런 불만이 여러 번 쌓이면 어느 순간 고객이 떠나거나 크게 화를 내는 위기로 번집니다. 이 저장소에는 두 가지가 들어 있습니다. 하나는 그 *쌓임*을 추적해 위기를 미리 알려 주는 부품(컴포넌트)이고, 다른 하나는 그 부품이 정말 효과가 있는지 직접 재 본 측정 결과입니다.

핵심 아이디어를 비유로 들면 이렇습니다. 보통의 안전장치는 *매번 처음 보는 검문소*와 같습니다. 지금 들어온 메시지 하나만 보고 통과시킬지 막을지 판단합니다(이런 단발 방식을 stateless라고 부릅니다). 이 프로젝트가 만든 것은 *단골의 행동을 기억하는 경비원*에 가깝습니다. 대화가 어떻게 흘러왔는지를 누적해서 보고, 약한 불만이라도 쌓이면 위기 신호로 잡아냅니다(이런 누적 방식을 stateful이라고 부릅니다). 부품은 대화가 진행될 때마다 위기 점수를 조금씩 더해 가는 엔진(`S_t = λ·S_{t-1} + signal_t`), 점수에 따라 봇→상담사→매니저로 넘기는 3단계 이관, 그리고 대화가 아무리 길어져도 한 턴 처리 비용이 일정하게 유지되는 구조(O(N))로 이뤄져 있고, `sgr escalate` 한 줄이면 바로 돌려볼 수 있습니다. 측정은 이 누적 방식이 단발 방식보다 위기를 더 일찍 잡는지를, 눈대중이 아니라 통계 검정(McNemar 검정·부트스트랩 신뢰구간·λ-sweep)으로 강한 비교 대상과 맞대 본 결과입니다. 결론부터 말하면, 한 가지 방식에서는 우위가 분명했고 다른 한 가지에서는 성립하지 않았으며, 더 잘 잡는 대신 멀쩡한 대화를 위기로 잘못 본 경우도 늘었습니다. 좋은 결과만 고르지 않고 안 된 것도 아래에 그대로 적었습니다.

## 배경 / 왜 만들었나

AI 비서(에이전트)가 실제 서비스에 들어가면서, 새로운 종류의 위협이 떠올랐습니다. 메시지 하나하나는 멀쩡해 보이지만, 여러 번에 걸쳐 조금씩 밀어붙여 AI가 하면 안 되는 행동을 하게 만드는 방식입니다(점진적 조작). 2026년 6월 메타 고객지원 챗봇 사고가 그런 사례입니다. 중요한 건 그 사고가 한 번에 일어났느냐가 아니라, "하나씩 보면 무해한데 쌓이면 위기가 된다"는 구조 자체가 매 메시지를 따로따로 보는 안전장치의 사각지대라는 점입니다.

이 프로젝트의 목표는 그런 누적형 위협을 *막는 것*이 아니라 *일찍 알아채는 것*입니다. 대화 전체를 누적해서 보는 방식(누적/stateful)이, 매 메시지만 보거나 최근 몇 개만 보는 방식(단발/stateless)보다 이상 징후를 더 일찍 포착하는지를, 틀렸으면 틀렸다고 드러나는 방식으로 측정하는 것이 목표였습니다.

조작이나 탈옥(jailbreak) 데이터를 직접 만들면 안전정책과 충돌합니다. 그래서 구조는 똑같으면서 안전한 대체 주제로 "고객지원 위기 이관"을 골랐습니다. 개별 불만은 하나로는 무해하지만 쌓이면 위기가 된다는 점에서 누적형 조작과 구조가 같고, 측정에 쓴 방법(신호 설계, 비교 대상 설정, 통계 검정, 작동점 정렬, 신뢰구간)은 주제를 가리지 않으므로 나중에 조작 탐지 쪽으로 옮겨 갈 여지도 열려 있습니다. 안전정책과 부딪힐 걸 미리 알고 우회한 판단이었습니다.

## 탐지에서 대응으로 — 운영 가치

위기를 탐지하는 것 자체로는 끝이 아니라, 그게 실제 *대응*으로 이어질 때 값이 있습니다. 아래 데모와 비용 비교로 이 부품을 어디에 쓸 수 있는지 보여 주고, 이어지는 측정 섹션에서 그 효과를 수치로 검증합니다. 데모는 가짜로 만든(합성) 대화이고 비용은 추정치라는 점을 미리 밝혀 둡니다.

### 에스컬레이션 3단계

누적 위기 점수 `S_t`를 미리 고정해 둔 두 임계값 `t1=0.7`, `t2=0.9`에 비춰 자동으로 처리 경로를 정합니다(재조정 없음). `S_t`가 `t1` 미만이면 봇이 자동 응대하고, `t1` 이상 `t2` 미만이면 상담사에게, `t2` 이상이면 매니저·이탈방지팀으로 넘깁니다.

```
$ sgr escalate --session c1-test-004

  턴   risk    S_t  STATEFUL 단계        B1(단발) 단계
    0   0.00   0.00  봇 자동응대            봇 자동응대
    1   0.42   0.42  봇 자동응대            봇 자동응대     ← "몇 번을 다시 로그인해도 튕겨요"
    2   0.55   0.84  상담사 이관            봇 자동응대     ← "사흘째 연락하는데 그대로네요. 답답해요"
    3   0.90   1.00  매니저·이탈방지팀 이관   매니저·이탈방지팀 이관  ← "탈퇴할게요"

[이관 권고]
  STATEFUL(누적): 상담사 이관 = 2번째 턴 / 매니저 이관 = 3번째 턴
  B1(stateless 단발): 상담사 이관 = 3번째 턴 / 매니저 이관 = 3번째 턴
  → 선제 이관: STATEFUL이 B1보다 1턴 일찍 상담사에 넘긴다
    (약한 불만 누적이 명시적 위기 선언 *이전에* 사람 개입을 유도)
```

단발 방식(B1)은 매 턴을 따로 보기 때문에 "언제 사람에게 넘길지"를 누적으로 판단하지 못합니다. 위 데모의 2번째 턴 "답답해요"는 그 한 줄만 보면 위기 점수가 0.55로 `t1`에 못 미쳐 봇 응대로 남지만, 지금까지 쌓인 누적 점수 `S_t=0.84`는 상담사 이관 임계를 넘깁니다. 그 결과 고객이 "탈퇴"를 입에 올리기 한 턴 전에 사람에게 넘어갑니다. 일찍 잡은 신호가 곧바로 대응으로 이어지는 지점입니다. 모든 판정은 `sgr audit --session <id>`로 나중에 추적할 수 있습니다(ISC-4.2).

다만 모든 위기 대화에서 이런 선제성이 나타나지는 않습니다. 약한 불만이 한 턴씩 띄엄띄엄 들어오면 λ=0.7로 점수가 서서히 식어 누적이 흩어지고, 단발과 이관 시점이 같아지는 대화도 있습니다. 이런 경우도 그대로 보고합니다.

### 경제성(비용)

누적 방식(STATEFUL)은 매 턴 숫자 하나(`S_{t-1}`)만 들고 가면 되므로, 대화가 길어져도 한 턴 처리 비용이 늘지 않습니다. 반면 B1.5는 최근 5턴(K=5)을, B2는 대화 전체를 매 턴 판정기(judge)에 다시 집어넣습니다. N턴짜리 대화에서 판정기에 들어가는 누적 토큰(처리량)을 비교하면 차이가 분명해집니다.

| N(세션 턴) | STATEFUL O(N) | B1.5 O(N·K) | B2 O(N²) | B1.5/ST | B2/ST |
|---|---|---|---|---|---|
| 10 | 210 | 760 | 1,045 | ×3.6 | ×5.0 |
| 20 | 420 | 1,710 | 3,990 | ×4.1 | ×9.5 |
| 50 | 1,050 | 4,560 | 24,225 | ×4.3 | ×23.1 |
| 100 | 2,100 | 9,310 | 95,950 | ×4.4 | ×45.7 |

표 읽는 법: 맨 오른쪽 두 칸이 누적 방식 대비 몇 배 비싼가입니다. 대화가 길어질수록(N이 커질수록) B2는 누적 대비 ×5.0 → ×45.7로 격차가 벌어집니다. 즉 긴 대화나 실시간 운영에서는 전체를 매번 다시 넣는 방식(B2)이 비용상 사실상 불가능하고, 최근 몇 턴만 넣는 방식(B1.5)도 K배가 듭니다. 측정 결론이 내세우는 "B2급 정확도를 B2의 1/N 비용으로"라는 정당성을 이 숫자가 받칩니다. 다만 이 표는 실제로 잰 게 아니라 추정입니다. 한 턴을 약 19개 토큰으로(동봉 데이터의 한국어 메시지 평균 약 28자 기준), 판정기 단가를 가정해 환산한 값입니다. 로컬 Ollama로 돌리면 실제 돈은 0원이고, 이 표는 클라우드 판정기로 옮겼을 때의 상대적 규모만 보여 줄 뿐입니다. 재현은 `sgr cost-model`, 자세한 내용은 [`out/cost.md`](out/cost.md)에 있습니다.

## 측정 결과

이제 효과 자체를 봅니다. 평가 원본 수치는 [`out/mini.md`](out/mini.md)에 있습니다. 비교는 위기가 실제로 일어난 대화 62개(C1 양성 test: c1.test 47 + c1.calib 합류 15)와 정상적으로 잘 해결된 대화 40개(C3 test, 그중 22개는 "길지만 정상"인 대조군)를 두고, 최근 5턴(K=5)을 보는 방식과 맞대 본 것입니다.

먼저 용어 두 개만 풀어 둡니다. **recall(재현율)**은 "실제 위기를 놓치지 않고 얼마나 잡아냈는가"의 비율이고, **오탐율(FPR)**은 "멀쩡한 대화를 위기로 잘못 본" 비율입니다. 좋은 위기 탐지기는 이 둘을 동시에 잘해야 합니다 — 위기를 많이 잡되, 멀쩡한 걸 잘못 잡지는 말아야 하죠.

핵심부터 말하면, `target_aware`라는 한 가지 탐지 방식에서 누적(STATEFUL)이 최근 5턴만 보는 단발(B1.5)을 뚜렷하게 이겼습니다.

| 비교 | Δrecall(전체) | Δrecall(K초과) | ΔTTD(전체) | McNemar p |
|---|---|---|---|---|
| STATEFUL − B1 (per-turn) | +19.4%p | +29.6%p | −0.02 | 0.004 |
| STATEFUL − B1.5 (sliding-window) | +53.2%p | +63.0%p | +0.30 | 0.000 |

표 읽는 법: 각 숫자는 "누적 방식이 단발 방식보다 얼마나 더 나았는가"의 차이(%p, 퍼센트포인트)이고, 양수면 누적이 더 좋다는 뜻입니다. 맨 오른쪽 McNemar p는 "이 차이가 우연일 확률"로, 작을수록(보통 0.05 미만) 우연이 아니라는 강한 증거입니다. ΔTTD는 "누적 방식이 단발보다 몇 턴 일찍 잡았는가"로, 양수면 더 일찍 잡았다는 뜻입니다(success_turn − detect_turn). 그리고 단발(B1.5) 대비 recall 차이를, 표본을 바꿔 가며 2000번 다시 계산해 봐도(부트스트랩 신뢰구간) 그 범위가 [+40.3%p, +66.1%p]로 전부 0보다 컸습니다 — 차이의 방향이 우연으로 뒤집히지 않는다는 뜻입니다.

이 우위가 "기준선을 느슨하게 잡아서 생긴 착시"가 아닌지도 따로 확인했습니다. 양쪽이 멀쩡한 대화를 잘못 잡는 비율(오탐율)을 똑같이 맞춰 놓고 recall을 다시 재도, 누적이 단발을 오탐 ≤5%에서 +19.4%p, ≤10%에서 +22.6%p, ≤20%에서 +38.7%p로 일관되게 앞섭니다. 또 핵심 설정값(λ)을 0.5·0.7·0.9·1.0으로 바꿔 가며 결론이 흔들리는지 봐도(λ-sweep), 단발(B1.5) 대비 차이는 +32~+53%p 사이로 한 번도 부호가 뒤집히지 않았습니다. "오탐을 허용해서 산 우위"라거나 "표본이 작아 생긴 우연"이라는 반박을, 작동점 정렬·신뢰구간·λ 전 구간이라는 세 겹으로 닫은 셈입니다.

대신 이 recall 우위는 공짜가 아니었습니다. "길지만 정상"인 긴 대화에서, 누적 방식이 멀쩡한 대화를 위기로 잘못 본 비율은 18%(22개 중 4개)로, 단발(B1.5)의 9%(22개 중 2개)보다 높았습니다. "대화가 길어서 점수가 오른 것"과 "실제로 위기라서 점수가 오른 것"을 끝까지 갈라내지 못한 대가입니다.

반대쪽 탐지 방식인 `target_agnostic`에서는 우위가 성립하지 않았습니다. 누적은 단발(B1) 대비 +0.0%p(p=1.000), 단발(B1.5) 대비 −4.8%p(p=0.250)에 그쳤고, 신뢰구간도 [−11.3%p, +0.0%p]로 0을 포함합니다(차이가 없을 수도 있다는 뜻). 다만 McNemar 검정은 두 방식의 판정이 엇갈린 경우만 세기 때문에, 표본이 작으면 우연과 구별하는 힘이 약합니다. "유의함"은 강한 증거지만, "유의하지 않음"이 곧 "효과가 없음의 증명"은 아닙니다.

이 비대칭은 오히려 설계가 의도대로 작동했다는 신호로 읽힙니다. `target_agnostic`은 "직전 몇 턴과 비교해 말의 방향이 얼마나 바뀌었나"만 봅니다. 그런데 이 양은 위기든 정상이든 비슷하게 커서, 위기를 가려내는 힘이 약합니다. 목표(위기)를 향해 쌓여 가는 움직임을 보는 쪽(`target_aware`)만 성립하고, 방향과 무관한 변화를 보는 쪽(`target_agnostic`)은 성립하지 않았다는 건, 데이터가 어느 한 탐지기에 유리하게 짜여 있지 않았다는 걸 부분적으로 뒷받침합니다. 다만 이게 완벽한 반대 통제는 아니어서, 결과가 데이터 설계 덕분에 저절로 나왔을 가능성(순환)을 완전히 배제하지는 못합니다.

## 한 줄 결론

`target_aware` 한 탐지 방식에서, 누적(STATEFUL)이 단발(stateless)을 — 오탐율을 맞춰도, 표본을 2000번 다시 뽑아도, 설정값을 바꿔도 — 견고하게 이겼습니다(McNemar p=0.000). 단, 순환 가능성을 걸러내려고 둔 대조군 `target_agnostic`에서는 성립하지 않아 순환 가능성을 완전히 배제하지 못하고, 위기를 더 잘 잡는 대신 긴 정상 대화를 위기로 잘못 본 비율이 18%로 늘었습니다. 유리하든 불리하든 수치를 그대로 적는 Mycelium의 `graph_weight=0.0` 정신을 잇습니다.

## 측정 규율

좋은 수치를 내는 것보다 "어떻게 측정했는가"가 이 프로젝트의 본체입니다. 그래서 측정을 "안 됐다고 정직하게 보고하기"에 그치지 않고, 틀렸으면 틀렸다고 드러나는 실험 설계로 다뤘습니다.

**공정한 비교 대상을 세웠습니다.** 비교 상대(baseline)는 셋입니다 — 매 메시지만 보는 단발(B1), 최근 5턴만 보는 단발(B1.5), 그리고 누적(STATEFUL). 셋 다 "멀쩡한 대화를 잘못 잡는 비율"을 똑같이 5%로 맞춰, "결국 최근 몇 턴 보는 거랑 같은 거 아니냐"는 반박에 정면으로 답하게 했습니다.

**설정값은 미리 못 박고 건드리지 않았습니다.** λ=0.7, K=5, 이관 임계 `t1`·`t2`를 모두 시험 데이터(test)를 보기 전에 별도의 보정용 데이터(calibration)에서만 정해 고정했고, 측정 도중에 다시 손대지 않았습니다(시험 데이터를 보고 설정을 맞추면 성적이 부풀려지기 때문입니다).

**대화 길이가 결과를 흐리지 않도록 통제했습니다.** 정상 대화(C3)의 길이 중앙값이 6턴으로 위기 대화(C1)의 5턴보다 길었습니다. 누적 방식은 대화가 길수록 점수가 올라가기 때문에, 자칫 "길어서" 위기로 잘못 잡을 수 있습니다. 그래서 "길지만 정상"인 대조군을 22개로 늘려 점검 힘을 키웠고, 최근 5턴 안에서 끝난 35개와 5턴을 넘긴 27개를 나눠서 보고했습니다. 5턴 이내에서는 단발(B1.5)과 누적이 구조적으로 거의 같다는 사실까지 그대로 공개합니다.

**결과가 저절로 나오게 짜이지 않았는지 점검했습니다.** 탐지 방식을 둘로 나눠 동시에 돌렸습니다 — 목표(위기)를 향해 쌓이는 움직임을 보는 `target_aware`와, 방향과 무관하게 직전 몇 턴 대비 말이 얼마나 달라졌나만 보는 `target_agnostic`. 한쪽만 성립하면, 데이터가 어느 한 탐지기에 유리하게 짜이지 않았다고 볼 근거가 됩니다. 데이터를 만든 AI와 평가하는 AI도 분리했습니다 — 데이터는 claude-opus(Anthropic)로 만들고, 평가·임베딩은 bge-m3와 qwen2.5:14b(Alibaba)로 처리해, 만든 쪽의 문체가 평가에 되돌아와 영향을 주는 위험을 줄였습니다.

**"이겼다"는 눈대중이 아니라 통계 검정으로 판정했습니다.** 가장 중요한 비교(단발 B1.5 대비)에서 "이 차이가 우연일 확률"(McNemar 검정)이 0.05 미만일 때만 "성립"으로 선언했고, 오탐율을 맞춘 재비교와 표본을 2000번 다시 뽑는 신뢰구간(seed 고정)으로 불확실성도 함께 드러냈습니다. 설정값(λ)도 0.5·0.7·0.9·1.0 전 구간에서 차이 곡선을 그려 봤고, 어디선가 부호가 뒤집혔다면 그대로 공개할 작정이었습니다. 그 결과는 [`out/lambda.md`](out/lambda.md)에 있습니다.

## 아키텍처

의존 방향은 `interfaces → pipeline → adapters → core`입니다. 바깥쪽(CLI·외부 연동)이 안쪽(순수 계산)에 의존하고 그 반대는 없는, Mycelium에서 이어받은 클린 아키텍처입니다.

```
interfaces/   CLI(typer, sgr 명령) — 입출력 진입점
pipeline/     판정 파이프라인 오케스트레이션
              (메시지 인입 → 정책 실행 → 에스컬레이션 결정 → 감사로그)
adapters/     외부 경계: LLM/임베딩 provider(Ollama), 상태 영속화(JSON), 감사로그
core/         순수 도메인: Policy 인터페이스, SessionState, 누적 판정 알고리즘
              (외부 I/O·프레임워크 의존 없음 — 수치 연산은 stdlib math만)
```

누적 공식은 `S_t = clip(λ·S_{t-1} + signal_t, 0, S_max)`로, 최근 신호일수록 더 무겁게 쳐서 위기 점수를 쌓되 오래된 신호는 λ=0.7 비율로 서서히 잊는 방식입니다(EWMA, 지수가중이동평균). 눈여겨볼 점은 비교 대상(baseline)이 따로 만든 코드가 아니라 같은 엔진의 다른 모드라는 것입니다. 과거 상태를 안 보면 B1(단발), 최근 K턴만 슬라이딩하면 B1.5, 누적 상태를 참조하면 STATEFUL이 됩니다. 같은 데이터·같은 정책 위에서 세 경로의 차이만 잰 셈입니다. 설계 결정(D-1~D-10)은 [`docs/DESIGN.md`](docs/DESIGN.md)에, 실행 계획과 검증 기준(ISC)은 [`docs/PLAN.md`](docs/PLAN.md)에 있습니다.

## 빠른 시작

먼저 저장소를 받아 가상환경을 만들고 개발 의존성을 설치합니다.

```bash
git clone <repo-url> stateful-guardrails
cd stateful-guardrails
python3.12 -m venv .venv
.venv/bin/pip install -e ".[dev]"
```

평가에는 Ollama와 모델 두 개가 필요합니다. 판정기 LLM(`qwen2.5:14b`)은 eval 단계에서만 쓰입니다.

```bash
# https://ollama.com 설치 후
ollama serve
ollama pull bge-m3        # 임베딩 (1.2 GB)
ollama pull qwen2.5:14b   # judge LLM (9 GB) — eval 시에만 필요
```

기본 명령은 다음과 같습니다. `sgr eval --mini`가 이 프로젝트의 사활이 걸린 핵심 검증입니다.

```bash
sgr --version                                          # 버전 확인 (ISC-0.1)
sgr catalog                                            # 정책 카탈로그 (category, stateless|stateful)
sgr scan --input data/c1.test.jsonl                    # 단발 스캔 (stateless 모드)
sgr eval --mini --dataset data/ --report out/mini.md  # 미니 평가 — 핵심 검증
```

측정을 재현하려면 동봉한 합성 데이터와 고정 파라미터로 평가를 돌린 뒤, 결과를 동봉된 `out/mini.md`와 비교하면 됩니다. λ-sweep도 같은 방식으로 재현됩니다(ISC-5.6).

```bash
sgr eval --mini --dataset data/ --report out/mini.md
sgr eval --mini --lambda-sweep 0.5,0.7,0.9,1.0 --report out/lambda.md --dataset data/
```

## 데이터

`data/` 디렉토리에는 합성(가짜로 만든) 대화 세션만 들어 있고 민감정보는 없습니다. `c1.{calib,test}.jsonl`은 누적 위기 에스컬레이션 세션(합성 한국어 포함), `c3.{calib,test}.jsonl`은 정상 해결 대화와 "길지만 정상"인 대조군 세션이며, `calibration.json`에 고정 파라미터(λ, K, N, S_max, 임계, FPR 예산)가 담겨 있습니다. 실제 고객 상담 로그는 포함하지 않습니다. 이 합성 데이터는 Mycelium의 `sample_vault/`와 같은 역할로, 클론하자마자 평가를 재현할 수 있도록 동봉했습니다.

Python 3.12 · typer · httpx · Ollama(`bge-m3` 임베딩 + `qwen2.5:14b` judge). 수치 연산은 외부 라이브러리 없이 stdlib `math`만 써서 core 레이어의 의존성을 최소화했습니다.
