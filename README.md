한국어 · [English](README.en.md)

대화형 AI에서 하나하나의 메시지는 사소한 불만에 그치지만, 그것이 여러 턴에 걸쳐 쌓이면 고객 이탈이나 격분 같은 위기로 번진다. 이 저장소에는 그 누적을 추적하는 위기 조기경보 컴포넌트와, 그 컴포넌트가 정말 효과가 있는지 따져 본 측정이 함께 들어 있다. 컴포넌트는 누적 위기 점수 엔진(`S_t = λ·S_{t-1} + signal_t`), 3단계 에스컬레이션 이관, 그리고 세션이 아무리 길어져도 턴당 비용이 일정한 O(N) 구조로 이뤄져 있으며 `sgr escalate` 한 줄로 바로 돌려볼 수 있다. 측정은 이 컴포넌트가 sliding-window 방식보다 위기를 더 일찍 잡아내는지를 강한 baseline 상대로 McNemar 검정과 부트스트랩 신뢰구간, λ-sweep으로 물은 결과다. 결론부터 말하면 한 detector에서는 우위가 분명히 성립했고 다른 한쪽에서는 성립하지 않았으며, 우위에는 오탐이라는 대가가 따랐다. 그 내용을 아래에 그대로 적는다.

## 배경 / 왜 만들었나

AI 에이전트가 실서비스에 들어가면서, 단발 가드레일이 무심코 통과시키는 점진적 조작이 현실적인 위협으로 떠올랐다. 개별 메시지는 무해해 보이지만 여러 턴에 걸쳐 쌓이면 AI가 경계를 넘거나 위험한 방향으로 끌려가는 패턴이다. 2026년 6월 메타 고객지원 챗봇 사고가 그런 사례다. 여기서 중요한 건 사고가 단발이었느냐 아니냐가 아니라, "개별로는 무해하지만 누적하면 위기가 된다"는 구조 자체가 매 턴을 따로 보는 가드레일의 사각지대라는 점이다.

우리가 시도한 건 이 누적형 위협을 막는 게 아니라 일찍 알아채는 것이다. 대화 이력 전체를 누적해 추적하는 stateful 방식이, 매 턴만 보거나 최근 몇 턴만 보는 stateless 방식보다 이상 징후를 더 이르게 포착하는지를 반증 가능한 형태로 측정하는 것이 목표였다.

조작이나 탈옥 데이터를 직접 만들면 안전정책과 충돌한다. 그래서 그와 구조가 같으면서도 안전한 대리 도메인으로 고객지원 위기 에스컬레이션을 택했다. 개별 불만 메시지는 단발로는 무해하지만 쌓이면 위기가 된다는 점에서 누적형 조작과 동형이고, 측정 골격(signal 설계, baseline 비교, McNemar, 작동점 정렬, 신뢰구간)은 도메인을 타지 않으므로 조작 탐지 쪽으로의 이론적 전이도 열려 있다. 안전정책과의 충돌을 미리 인지하고 우회한 판단이었다.

## 탐지가 만들어내는 것 — 운영 가치

탐지는 그 자체로 끝이 아니라 액션으로 이어질 때 값이 있다. 아래 데모와 비용 비교는 컴포넌트가 무엇을 가능하게 하는지를 보이고, 그다음 측정 섹션이 그 효과를 수치로 검증한다. 데모는 합성 세션이고 비용은 모델 추정이라는 점을 미리 밝혀 둔다.

### 에스컬레이션 3단계

누적 위기 점수 `S_t`를 동결 임계 `t1=0.7`, `t2=0.9`에 매핑해 라우팅한다. 재튜닝은 없다. `S_t`가 `t1` 미만이면 봇이 자동 응대하고, `t1` 이상 `t2` 미만이면 상담사에게, `t2` 이상이면 매니저·이탈방지팀으로 넘긴다.

```
$ sgr escalate --session c1-test-004

  턴   risk    S_t  STATEFUL 단계        B1(단발) 단계
    0   0.00   0.00  봇 자동응대            봇 자동응대
    1   0.42   0.42  봇 자동응대            봇 자동응대     ← "몇 번을 다시 로그인해도 튕겨요"
    2   0.55   0.84  상담사 이관            봇 자동응대     ← "사흘째 연락하는데 그대로네요. 답답해요"
    3   0.90   1.00  매니저·이탈방지팀 이관   매니저·이탈방지팀 이관  ← "탈퇴할게요"

[이관 권고]
  STATEFUL(누적): 상담사 이관 = 2번째 턴 / 매니저 이관 = 3번째 턴
  B1(stateless 단발): 상담사 이관 = 3번째 턴 / 매니저 이관 = 3번째 턴
  → 선제 이관: STATEFUL이 B1보다 1턴 일찍 상담사에 넘긴다
    (약한 불만 누적이 명시적 위기 선언 *이전에* 사람 개입을 유도)
```

stateless 단발(B1)은 매 턴을 독립으로 보기 때문에 "언제 사람에게 넘길지"를 누적으로 판단하지 못한다. 위 데모의 2번째 턴 "답답해요"는 단발 risk가 0.55로 `t1`에 못 미쳐 봇 응대로 남지만, 누적 점수 `S_t=0.84`는 상담사 이관 임계를 넘긴다. 그 결과 고객이 "탈퇴"를 입에 올리기 한 턴 전에 사람에게 넘어간다. 조기 탐지가 곧 가능해지는 액션인 셈이다. 모든 판정은 `sgr audit --session <id>`로 감사 추적된다(ISC-4.2).

다만 모든 C1 세션에서 이런 선제성이 나타나지는 않는다. 약한 불만이 한 턴씩 띄엄띄엄 들어오면 λ=0.7 감쇠 때문에 누적이 흩어져, 단발과 이관 시점이 같아지는 세션도 있다. 이런 경우도 그대로 보고한다.

### 경제성

STATEFUL은 매 턴 스칼라 하나(`S_{t-1}`)만 참조하므로 턴당 비용이 고정이다. 반면 B1.5는 최근 K=5턴을, B2는 세션 전체를 매 턴 judge에 다시 밀어 넣는다. N턴 세션에 누적되는 judge 입력 토큰을 비교하면 차이가 분명해진다.

| N(세션 턴) | STATEFUL O(N) | B1.5 O(N·K) | B2 O(N²) | B1.5/ST | B2/ST |
|---|---|---|---|---|---|
| 10 | 210 | 760 | 1,045 | ×3.6 | ×5.0 |
| 20 | 420 | 1,710 | 3,990 | ×4.1 | ×9.5 |
| 50 | 1,050 | 4,560 | 24,225 | ×4.3 | ×23.1 |
| 100 | 2,100 | 9,310 | 95,950 | ×4.4 | ×45.7 |

B2 대비 배수가 N에 비례해 단조 증가하는데(O(N²) 대 O(N)), 긴 세션이나 online 운영에서는 전체 재투입(B2)이 비용상 사실상 불가능하고 B1.5도 K배가 든다는 뜻이다. 측정 결론이 내세우는 "B2급 정확도를 B2의 1/N 비용으로"라는 정당성을 이 숫자가 받친다. 다만 이 표는 실측이 아니라 모델 추정이다. 턴당 토큰을 약 19개로(동봉 데이터의 한국어 user 메시지 평균 약 28자 기준), judge 단가를 가정해 환산한 값이다. 로컬 Ollama의 실제 금전 비용은 0이고, 표는 클라우드 judge로 옮겼을 때의 상대 규모를 보여 줄 뿐이다. 재현은 `sgr cost-model`, 상세는 [`out/cost.md`](out/cost.md)에 있다.

## 측정 결과

이제 효과 자체를 본다. 평가 원본은 [`out/mini.md`](out/mini.md)에 있고, C1(누적 위기) 양성 test 62세션(c1.test 47 + c1.calib 합류 15)과 C3(정상 해결) test 40세션(장기-양성 22 포함)을 K=5로 비교했다.

핵심부터 말하면, `target_aware` detector에서 STATEFUL은 sliding-window(B1.5)를 뚜렷하게 이겼다.

| 비교 | Δrecall(전체) | Δrecall(K초과) | ΔTTD(전체) | McNemar p |
|---|---|---|---|---|
| STATEFUL − B1 (per-turn) | +19.4%p | +29.6%p | −0.02 | 0.004 |
| STATEFUL − B1.5 (sliding-window) | +53.2%p | +63.0%p | +0.30 | 0.000 |

여기서 TTD는 success_turn에서 detect_turn을 뺀 값으로, 양수면 STATEFUL이 더 이른 턴에 탐지했다는 뜻이다. B1.5 대비 Δrecall의 부트스트랩 95% 신뢰구간은 [+40.3%p, +66.1%p]로 하한이 0을 넘으므로 부호가 확실하다.

이 우위가 작동점이 달라서 생긴 착시가 아니라는 것도 따로 확인했다. 같은 test-FPR 예산으로 임계를 맞춰 recall을 다시 재면 ≤5%에서 +19.4%p, ≤10%에서 +22.6%p, ≤20%에서 +38.7%p로 STATEFUL이 B1.5를 일관되게 앞선다. λ를 0.5·0.7·0.9·1.0으로 훑어도 B1.5 대비 델타는 +32~+53%p 사이에서 부호가 한 번도 뒤집히지 않는다. "FPR을 주고 산 우위"나 "소표본 노이즈"라는 반론을 작동점 정렬·신뢰구간·λ 전구간이라는 세 겹으로 닫은 셈이다.

대신 이 recall 우위는 공짜가 아니다. 길이 교란을 통제한 장기-양성 세션에서 STATEFUL의 오탐율은 18%(4/22)로, B1.5의 9%(2/22)보다 높다. '긴 세션이라 위기'와 '에스컬레이션이라 위기'를 끝까지 갈라내지 못한 대가다.

반대편 detector인 `target_agnostic`에서는 우위가 성립하지 않았다. STATEFUL은 B1 대비 +0.0%p(p=1.000), B1.5 대비 −4.8%p(p=0.250)에 그쳤고 신뢰구간도 [−11.3%p, +0.0%p]로 0을 포함한다. McNemar는 불일치쌍에만 의존하므로 표본이 작으면 검정력이 낮다는 점은 감안해야 한다. 유의는 강한 증거이지만, 비유의가 곧 반증은 아니다.

이 비대칭은 오히려 설계가 의도대로 작동했다는 신호로 읽힌다. `target_agnostic`은 직전 N턴 대비 의미 이동량만 보는데, 이 양은 위기 세션이든 정상 세션이든 비슷하게 커서 위기 신호로서의 분리력이 약하다. 목표 개념을 향한 누적 이동(`target_aware`) 한쪽만 성립하고 목표 무관 드리프트(`target_agnostic`)가 성립하지 않았다는 건, 데이터 설계가 탐지축으로 새지 않았음을 부분적으로 뒷받침한다. 다만 이것이 완전한 역통제는 아니어서 순환 가능성을 전부 배제하지는 못한다.

## 한 줄 결론

`target_aware` 한 detector에서 STATEFUL이 stateless를 작동점 정렬·신뢰구간·λ 전구간에 걸쳐 견고하게 이겼다(McNemar p=0.000). 단 비순환 통제군인 `target_agnostic`은 성립하지 않아 순환 가능성을 완전히 배제하지 못하고, recall 우위는 장기정상 오탐 18%라는 비용을 동반한다. 유불리와 무관하게 수치를 그대로 적는 Mycelium의 `graph_weight=0.0` 정신을 잇는다.

## 측정 규율

좋은 수치보다 어떻게 측정했는가가 이 프로젝트의 본체다. 그래서 측정 골격을 단순한 정직 negative 보고가 아니라 반증 가능한 실험 설계로 다뤘다.

비교는 공정한 강 baseline 위에서 했다. per-turn stateless(B1), sliding-window K=5(B1.5), STATEFUL을 동일 FPR 예산 5%로 맞춰 "결국 sliding-window의 손실 압축 아니냐"는 반론에 정면으로 답하게 했다. 파라미터는 calibration-split에서만 정하고 동결했다. λ=0.7, K=5, 임계 `t1`·`t2`를 모두 test-split을 보기 전에 고정했고 측정 중 재튜닝은 없다.

길이 교란은 따로 탐지하고 통제했다. C3 정상 세션의 중앙값이 6턴으로 C1 위기 세션의 5턴보다 길어, 누적식 특성상 점수가 길이에 비례해 부풀 수 있었다. 그래서 '긴데 정상'인 장기-양성 대조군을 22개로 확대해 통제 검정력을 보강했고, K 이내 35세션과 K 초과 27세션을 나눠 보고했다. K 이내에서는 B1.5와 STATEFUL이 구조적으로 거의 같다는 사실까지 그대로 공개한다.

자기충족을 막기 위해 detector를 둘로 나눠 병렬 비교했다. 목표 개념을 향한 누적 코사인 이동(`target_aware`)과 직전 N턴 대비 의미 이동량(`target_agnostic`)을 함께 돌려, 한쪽만 성립하면 데이터 설계가 탐지축으로 새지 않았다고 볼 근거로 삼았다. 생성과 평가 모델도 분리했다. 데이터는 claude-opus(Anthropic)로 만들고 평가·임베딩은 bge-m3와 qwen2.5:14b(Alibaba)로 처리해 생성 문체가 평가로 순환하는 위험을 줄였다.

성립 판정은 점추정 부호가 아니라 검정으로 했다. 사활 컬럼(vs B1.5)의 McNemar exact p가 0.05 미만일 때만 성립으로 선언하고, 작동점을 맞춘 recall 재비교와 seed를 고정한 2000회 부트스트랩 신뢰구간으로 불확실성을 함께 드러냈다. λ도 0.5·0.7·0.9·1.0 전 구간에서 델타 곡선을 산출했고, 부호가 뒤집히면 그대로 게시할 작정이었다. 그 결과는 [`out/lambda.md`](out/lambda.md)에 있다.

## 아키텍처

의존 방향은 `interfaces → pipeline → adapters → core`로, Mycelium에서 이어받은 클린 아키텍처다.

```
interfaces/   CLI(typer, sgr 명령) — 입출력 진입점
pipeline/     판정 파이프라인 오케스트레이션
              (메시지 인입 → 정책 실행 → 에스컬레이션 결정 → 감사로그)
adapters/     외부 경계: LLM/임베딩 provider(Ollama), 상태 영속화(JSON), 감사로그
core/         순수 도메인: Policy 인터페이스, SessionState, 누적 판정 알고리즘
              (외부 I/O·프레임워크 의존 없음 — 수치 연산은 stdlib math만)
```

누적 공식은 `S_t = clip(λ·S_{t-1} + signal_t, 0, S_max)`이고 λ=0.7의 지수 감쇠다. 눈여겨볼 점은 baseline이 별도 코드가 아니라 같은 엔진의 한 모드라는 것이다. `session_state=None`이면 B1(per-turn), 최근 K턴만 슬라이딩하면 B1.5, 누적 상태를 참조하면 STATEFUL이 된다. 같은 데이터와 같은 정책 위에서 세 경로의 델타만 잰다. 설계 결정(D-1~D-10)은 [`docs/DESIGN.md`](docs/DESIGN.md)에, 실행 계획과 검증 ISC는 [`docs/PLAN.md`](docs/PLAN.md)에 있다.

## 빠른 시작

먼저 저장소를 받아 가상환경을 만들고 개발 의존성을 설치한다.

```bash
git clone <repo-url> stateful-guardrails
cd stateful-guardrails
python3.12 -m venv .venv
.venv/bin/pip install -e ".[dev]"
```

평가에는 Ollama와 모델 두 개가 필요하다. judge LLM(`qwen2.5:14b`)은 eval 단계에서만 쓰인다.

```bash
# https://ollama.com 설치 후
ollama serve
ollama pull bge-m3        # 임베딩 (1.2 GB)
ollama pull qwen2.5:14b   # judge LLM (9 GB) — eval 시에만 필요
```

기본 명령은 다음과 같다. `sgr eval --mini`가 thesis의 사활점이다.

```bash
sgr --version                                          # 버전 확인 (ISC-0.1)
sgr catalog                                            # 정책 카탈로그 (category, stateless|stateful)
sgr scan --input data/c1.test.jsonl                    # 단발 스캔 (stateless 모드)
sgr eval --mini --dataset data/ --report out/mini.md  # 미니 평가 — thesis 사활점
```

측정을 재현하려면 동봉 합성 데이터와 동결 파라미터로 평가를 돌린 뒤 결과를 동봉 `out/mini.md`와 비교한다. λ-sweep도 같은 방식으로 재현된다(ISC-5.6).

```bash
sgr eval --mini --dataset data/ --report out/mini.md
sgr eval --mini --lambda-sweep 0.5,0.7,0.9,1.0 --report out/lambda.md --dataset data/
```

## 데이터

`data/` 디렉토리에는 합성 대화 세션만 들어 있고 민감정보는 없다. `c1.{calib,test}.jsonl`은 누적 위기 에스컬레이션 세션(합성 한국어 포함)이고, `c3.{calib,test}.jsonl`은 정상 해결 대화와 장기-양성 대조군 세션이며, `calibration.json`에 동결 파라미터(λ, K, N, S_max, 임계, FPR 예산)가 담겨 있다. 실제 고객 상담 로그는 포함하지 않는다. 이 합성 데이터는 Mycelium의 `sample_vault/`와 같은 역할로, 클론하자마자 평가를 재현할 수 있도록 동봉했다.
